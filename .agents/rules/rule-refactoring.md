# 🛠️ 코드 리팩토링 가이드라인

리팩토링의 핵심은 단순히 코드 라인 수를 줄이거나 중복 코드를 합치는 것(DRY)만이 아니라, **시스템의 전체적인 의도를 드러내고 개발자의 인지 부하를 줄이는 것**입니다. 리팩토링 작업을 시작할 때는 항상 다음의 원칙들을 적용해야 합니다.

## 1. 전역 규칙(Rule)들의 통합적 점검 및 적용

- 리팩토링을 시작하기 전, 작성된 모든 `rule-*.md` 파일의 내용을 숙지해야 합니다.
- 단순히 코드 레벨만 고치는 것이 아니라, 컴포넌트 구조(`rule-component-structure.md`), 컴포넌트 내부 로직 흐름(`rule-component-logic.md`), 훅의 부수 효과 제거(`rule-no-useeffect.md`), 네이밍 규칙(`rule-naming.md`) 등 **다른 규칙들이 잘 지켜져 있는지 통합해서 점검(Decision)** 하고 그 규칙을 기반으로 리팩토링을 수행해야 합니다.

## 2. 상위 레이어 맥락 파악 (Contextual Refactoring)

- 특정 파일이나 함수 단위로 좁게 리팩토링을 진행하지 마세요.
- **파일 계층에서의 부모 레벨(상위 컨테이너, 페이지 컴포넌트, 의존성 트리상 상위 모듈)의 맥락을 반드시 먼저 살펴봐야 합니다.**
- 하위 컴포넌트를 리팩토링할 때, 그 상위에서 어떤 형태의 Props를 내려주고 어떤 상태를 소비하는지를 알아야 근본적인 결합도(Coupling)를 낮출 수 있습니다.

## 3. 선언적 프로그래밍과 추상화 패턴 발굴

- **"어떻게(How)" 상태를 바꾸는지보다, "무엇(What)"과 데이터 상호작용의 "관계"에 집중하세요.**
- 제어 흐름(if, for, 중간 임시 변수 할당)을 캡슐화하고 데이터와 UI 간의 매핑 구조를 드러내야 합니다 (예: `SwitchCase` 패턴 사용 등).
- **공통화된 패턴 고려 (`overlay-kit` 사례 참고):** 단순히 컴포넌트 내부에서 함수 추출 수준을 넘어, 프로젝트에서 반복적으로 발생하는 디자인 패턴을 추상화하여 제공할 수 있는지 확인하세요.
  - 예시: 여러 곳에서 `isDialogVisible`, `openDialog()` 형태의 중복 상태로 오버레이를 닫고 여는 절차적인 코드라면, 이를 `overlay-kit` 처럼 **관계 기반**의 선언적 오버레이 호출 패턴으로 바꿀 수 있을지 고려해야 합니다.

## 4. 인지적 가독성을 높이는 뇌과학 원칙 (Neuroscience & Readable Code)

- **작업 기억(Working Memory)의 한계:** 인간의 뇌는 단기 기억의 슬롯이 4개 정도뿐입니다. 하나의 함수, 혹은 단일 컴포넌트 구조 안에서 개발자가 동시에 머리에 들고 추적해야 할 변수와 상태의 개수를 최소화하세요.
- **청킹(Chunking, 덩어리 직기):** 연관된 로직, 상태 변수, 관련 훅 등을 게슈탈트 원리(근접성의 원리, 유사성의 원리)에 맞게 물리적으로 응집력 있게 배치(그룹화)하세요. (예: `rule-component-logic.md`의 `useSetStateSelector` 그룹화 방식)
- **예측 가능성 향상 (예측 부호화, Predictive Coding):** 뇌는 코드를 글자대로 읽는 것이 아니라, 이전에 가지고 있던 패턴으로 '예측'합니다. 파일명, 변수명, 폴더 구조만 보아도 이후 전개될 로직의 흐름이 직관적으로(System 1 직관) 예측 가능하도록 프로젝트의 기존 코딩 스탠다드나 관례를 정확히 따르세요.

## 5. 💡 의도 파악과 프롬프트 깎기 (Interactive Prompt Shaping) [필수]

사용자가 **"이 컴포넌트 리팩토링 해줘"** 또는 **"코드 좀 정리해 줘"** 와 같이 **추상적인(Abstract) 리팩토링 요청**을 할 경우, AI는 단독으로 임의의 판단을 내려 코드를 수정하기 전에 **반드시 사용자에게 다시 질문하여 리팩토링의 구체적인 목표와 방향을 합의(Prompt Shaping)** 해야 합니다. 사용자가 구체적인 룰과 방향을 매번 다 짚어줄 수는 없으므로, AI가 주도적으로 아래와 같은 예시를 들어 질문하고 계획을 수립하세요.

**AI의 질문 예시 가이드라인:**

- _"어떤 방향의 개선을 원하시나요? 이 컴포넌트의 자식 컴포넌트 Props 드릴링을 줄이기 위해 Render Props나 Context API 패턴을 사용할까요, 아니면 내부 로직(`rule-component-logic.md`)의 방대한 상태를 청킹(Chunking)하는 방향으로 묶어드릴까요?"_
- _"오버레이 관련된 로직이 중복되는 것 같습니다. 기존 `overlay-kit`과 같이 선언적인 패턴(Declarative Pattern)으로 추상화하는 방향으로 리팩토링을 진행해볼까요?"_
- _"현재 파일 뿐만 아니라, 상위 레이어의 부모 컴포넌트부터 Props 구조를 변경하여 결합도(Coupling)를 낮추는 방향으로 전체적인 설계를 변경해볼까요?"_

**규칙:** AI는 사용자의 막연한 리팩토링 요청을 받으면, **반드시 X(현재의 문제 혹은 막연함)에서 O(명확한 리팩토링 목표/패턴)로 도달할 수 있도록 질문을 던지고 합의된 계획(Plan)을 바탕으로 코드를 수정**해야 합니다.
